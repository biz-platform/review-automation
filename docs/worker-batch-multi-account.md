# 워커 배치 처리 시 여러 계정(다중 user) 고려 사항

## 상황

워커는 **한 프로세스**에서 여러 사용자(계정)의 job을 순차적으로 처리한다.  
동시에:
- A 계정: 매장 S1 배민 댓글 3건
- B 계정: 매장 S2 배민 댓글 2건  
처럼 서로 다른 계정에서 다중 요청이 들어올 수 있다.

## 원칙

1. **한 배치 = 한 계정(한 user)**  
   같은 브라우저 세션에서 처리하는 배치는 **반드시 동일 `user_id`** 의 job만 포함해야 한다.  
   다른 계정의 job이 같은 배치에 섞이면, 잘못된 계정의 세션/쿠키로 요청이 나갈 수 있어 보안·정합성 문제가 된다.

2. **배치 선점 조건**  
   `claim_next_browser_job_batch` (또는 동일 역할 RPC)에서 “같은 그룹”으로 묶을 때 다음을 **모두** 만족하는 job만 추가한다.
   - `store_id` = 첫 번째로 선점한 job과 동일  
   - `type` = 첫 번째 job과 동일  
   - **`user_id` = 첫 번째 job과 동일**  

   즉, 배치 키는 **(store_id, type, user_id)** 이다.  
   (현재 스키마에서는 store가 한 user에 속하므로 `store_id`만으로도 user가 유일하지만, **명시적으로 user_id를 조건에 넣어** 다른 계정이 절대 섞이지 않도록 한다.)

3. **워커는 배치 간 상태를 넘기지 않음**  
   - 한 배치를 처리한 뒤에는 **반드시 브라우저(및 세션)를 종료**하고, 해당 배치의 job들에 대해만 결과 제출한다.  
   - 다음 루프에서 새로 claim한 배치는 **다른 user/다른 store**일 수 있으므로, 이전 배치의 브라우저·쿠키·세션을 재사용하지 않는다.

## 구현 체크리스트

- [ ] **DB RPC**  
  `claim_next_browser_job_batch` 에서 추가로 선점하는 행의 조건에  
  `user_id = (첫 번째로 선점한 job의 user_id)` 를 포함한다.

- [ ] **워커**  
  - 배치 처리 종료 시 브라우저/세션 종료를 한 번만 수행하고,  
    다음 `claim` 전에 이전 배치의 참조가 남지 않도록 한다.  
  - 배치 내부에서는 “첫 job의 user_id/store_id” 기준으로만 로그인·세션을 사용한다.

- [ ] **결과 제출**  
  각 job의 결과는 해당 job의 `id`·`user_id`·`store_id`에 맞는 API로만 제출한다.  
  (기존 `POST .../jobs/[id]/result` 가 job id로 구분하므로, job별로 제출하면 자동으로 올바른 계정/매장에 반영된다.)

이렇게 하면 **여러 계정에서 다중 요청이 들어와도**, 워커는 “한 배치 = 한 계정의 같은 매장·같은 타입”만 묶어서 한 브라우저에서 순차 처리하고, 계정 간 세션/요청이 섞이지 않는다.
