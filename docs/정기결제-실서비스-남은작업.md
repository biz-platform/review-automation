# 정기결제(월 빌링 / KORPAY MRB) 실서비스 체크리스트

> 목적: **일회성 수기결제 없이** 월정기결제(MRB)만으로 구독을 운영할 때, 프로덕션에 올리기 전/후에 해야 할 일을 한 곳에 모은다.  
> 전제: 카드 원문은 **가능하면 PG(Hosted)에서만** 입력받고, 서버는 **bid/tid/스케줄/금액/멱등키** 중심으로 운영한다.

---

## 이미 코드베이스에 있는 것 (기준점)

- **UI**
  - 요금제 선택 `/manage/billing/plan` (업/다운 UI, 차액 계산 표시)
  - 결제 관리 `/manage/billing/*` 일부
- **DB / 상태**
  - `public.users`: `paid_at`, `paid_until`, `cancel_at_period_end`, (마이그레이션 적용 시) `billing_pending_plan_key`, `billing_pending_plan_effective_at`
  - `public.member_billing_invoices`: 청구 이력(완료/오류, 이용 상태)
- **API (앱 서버)**
  - `GET /api/me/billing` (+ pending 필드)
  - `POST /api/me/billing/plan/upgrade` (프로→프리미엄: **서버 금액 검증 + DB 반영**; PG 승인은 아직 없음)
  - `POST /api/me/billing/plan/downgrade` (프리미엄→프로: **다음 주기 예약 플래그**)
  - `POST /api/me/billing/plan/pending/cancel`
- **문서**
  - `docs/KORPAY 월정기결제(빌링) 정리.md`

## 반드시 해야 하는 것 (실서비스 “정기결제”의 본체)

### 1) KORPAY MRB 연동 (핵심)

- **빌키 발급**: `POST /api/bill/new` (또는 PG가 제공하는 hosted/tokenize 플로우가 있으면 그쪽 우선)
- **재결제(청구)**: `POST /api/bill/approve` (멱등/중복 청구 방지)
- **전액 취소**: `POST /api/bill/cancel` (정책/시간 제한 확인)
- **빌키 삭제**: `DELETE /api/bill`
- **(선택) PG 스케줄**: `POST/DELETE /api/bill/schedule`, `POST /api/bill/schedule/list`  
  - “우리 DB 스케줄” vs “PG 스케줄” 중 하나로 **단일 원천**을 정해야 함(이중 예약 금지).

### 2) PCI / 민감정보 취급 (운영 리스크 최상)

- **카드번호/비밀번호/유효기간/주민(앞)/생년월일** 등은 로그·DB·에러 payload에 남기지 않기
- 서버가 원문을 받아야 하는 구조면:
  - TLS, 최소 보관(불가), 접근통제, 감사로그, 침해 대응 플랜까지 포함해 **현실적인 책임/비용** 평가
- 가능하면 **PG hosted + token/bid 중심**으로 설계하고, “우리 서버가 원문을 만지는 경로”를 제거

### 3) 월 청구 오케스트레이션 (지금 레포에서 가장 큰 빈칸)

> 현재 레포에는 “매월 자동으로 approve를 때리는” **단일 소스의 빌링 워커/크론 루프**가 아직 없다고 보는 게 안전하다.

- **대상 선정 쿼리**: `paid_until` 임박, `cancel_at_period_end`, `billing_pending_plan_key`, 결제수단(bid) 존재 여부 등
- **실패 재시도/Dead letter**: PG 응답 코드별 정책(재시도 간격, 최종 실패 처리)
- **성공 시 DB 반영 트랜잭션**
  - `users.paid_at/paid_until` 갱신
  - `member_billing_invoices` 신규 row 생성(다음 이용기간)
  - **다운그레이드 예약 소비**: 주기 시작 시점에 `plan_name`/금액/스케줄을 프로로 전환
- **운영 안전장치**
  - 크론 secret, 락(동시 실행 방지), 샤딩/파티션은 후순위지만 **중복 청구 방지**는 1순위

### 4) 업그레이드(차액) “PG 승인 → DB 커밋” 순서

- 지금 구현은 **DB 먼저 반영**에 가깝다. PG 연동 시에는 보통:
  - **승인 성공 응답을 받은 뒤에만** invoice/plan 갱신(원자성)
  - 실패/타임아웃 시 **보상 트랜잭션**(역청구/원복) 정책까지 포함

### 5) 웹훅/비동기 확정 (PG가 제공하면)

- 결제 결과가 비동기로 확정되는 모델이면 **웹훅 서명 검증 + 멱등 처리** 필수
- “요청은 성공인데 실제는 실패” 케이스를 UI/DB와 맞추기

### 6) 운영 필수

- **관측성**: 결제 시도/성공/실패/재시도 메트릭, 알림(슬랙 등)
- **CS 도구**: 특정 유저 billing 상태를 안전하게 조회/수정하는 어드민 플로우(감사로그)
- **법/약관/영수증**: 세금계산서/현금영수증 정책이 필요하면 별도 이슈

## “다음 작업(이전에 말한 A: PG 승인 후 DB 커밋)”은 언제 쓰나?

**정기결제(재청구) + 차액 결제(업그레이드)**까지 KORPAY `bill/approve`를 붙이는 단계에서 적용하면 된다.  
지금은 **상태/예약/화면/서버 검증** 레이어가 먼저 깔린 상태이고, PG 연동이 들어오면 **트랜잭션 경계**를 그때 맞추는 게 비용 대비 정답에 가깝다.
